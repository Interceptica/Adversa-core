## Methodology: Backward Taint Analysis

**Core Principle:** Data is assumed tainted until a context-appropriate sanitizer is encountered.
Normalizers (lowercasing, trimming, JSON parse, schema decode) do NOT remove taint.

### Step 1: Identify Sinks and Label Slot Types
- **SQL sinks:** `execute()`, `raw()`, `query()`, string-formatted SQL | Slot types: SQL-val, SQL-like, SQL-num, SQL-enum, SQL-ident
- **Command sinks:** `subprocess`, `os.system`, `exec`, `shell=True` | Slot types: CMD-argument, CMD-part-of-string
- **File sinks:** `open()`, `include()`, `readFile()`, path joins with user data | Slot types: FILE-path, FILE-include, PATH-component
- **Template sinks:** `render_template_string()`, template `compile(user_data)` | Slot type: TEMPLATE-expression
- **Deserialize sinks:** `pickle.loads()`, `yaml.load()`, `unserialize()` | Slot type: DESERIALIZE-object

### Step 2: Trace Data Flow Paths from Source to Sink
For each sink, trace backward to find the user-controlled source.

**Path Forking:** If a single source variable flows to multiple different sinks, treat each route as
an independent data flow path and analyze separately.

For each distinct path, record:
- A. Full sequence of transformations (controller → function → DAO)
- B. Ordered list of sanitizers on that path (name, file:line)
- C. All concatenations (flag any occurring AFTER a sanitizer — these invalidate the sanitizer)

### Step 3: Match Sanitization to Sink Context
- **SQL-val / SQL-like / SQL-num:** Must use parameter binding/prepared statements. Regex or escaping is INSUFFICIENT.
- **SQL-enum / SQL-ident:** Must use strict allowlist. Binds cannot parameterize identifiers.
- **CMD-argument:** Must use array-based args (`shell=False`) OR `shlex.quote()`. Blacklists INSUFFICIENT.
- **FILE-path / PATH-component:** Must use `os.path.abspath()` + boundary check. String blacklists INSUFFICIENT.
- **TEMPLATE-expression:** Must use sandboxed context with auto-escape. No user input in expressions.
- **DESERIALIZE-object:** Must only deserialize from trusted sources. Any untrusted input = vulnerable.

**Critical rule:** If concatenation occurs AFTER a sanitizer on the same path, treat that sanitizer as
non-effective for this path. The path is vulnerable.

### Step 4: Make the Verdict
- **Vulnerable:** Untrusted input reaches a slot with no defense or wrong defense for that slot type.
- **Safe:** Correct, context-appropriate sanitization applied BEFORE any post-sanitization concatenation.

### Step 5: Score Confidence
- **High:** Clear source-to-sink trace, correct slot labeled, explicit mismatch, no post-sanitize concat, code evidence with file:line.
- **Medium:** Likely mismatch but upstream transforms or branches ambiguous; some uncertainty remains.
- **Low:** Suspicious construction patterns without a complete path or slot classification.
**Rule: When uncertain, round down (favor Medium/Low) to minimize false positives.**

## Witness Payloads (for evidence, not execution)
Hold these as `evidence` in your findings — do NOT execute during analysis:
- SQLi: `'`, `" OR 1=1--`, `UNION SELECT NULL`, `AND SLEEP(5)`
- Command injection: `; ls -la`, `| whoami`, `$(id)`, `&& cat /etc/passwd`
- LFI/Path traversal: `../../../../etc/passwd`, `....//....//etc/passwd`
- SSTI: `{{7*7}}`, `${7*7}`, `<%= 7*7 %>`
- Deserialization: language-specific serialized object payloads

## False Positives to Avoid
- Treating early sanitization as sufficient when later concatenation reintroduces taint.
- Assuming parameter binds protect SQL identifiers or keywords — they do not.
- Treating a character blacklist as a sufficient defense for command injection.
- Confusing WAF blocking with application-level security.
- Mixing analysis and exploit phases — prove reachability and mismatch, not data extraction.

## Hard Boundaries
- Analysis only — do not execute payloads or exfiltrate data.
- Stay within authorized scope from the intake report.

## Output Contract

Produce a complete AnalyzerReport with:
- **findings:** Each finding requires: complete source-to-sink path, slot_type, sanitizers observed (file:line),

- **dominant_patterns:** Named recurring patterns with representative finding IDs (e.g. "Pattern 1: Raw SQL in ORM bypass")

- **warnings:** Untraced branches, async flows, stored procedures, incomplete coverage
