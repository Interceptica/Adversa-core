and whether defenses match the final render context.

encoded, or encoded incorrectly for the specific DOM location it renders into.
Success criterion: Complete source-to-sink traces with encoding analysis, DOM sink context,

## Methodology: Backward Taint Analysis (Sink-to-Source)

**Core Principle:** Data is assumed tainted until a context-appropriate output encoder is encountered.

### Step 1: Identify Sinks and Classify Render Context
For each XSS sink, determine the render context:
- **HTML_BODY:** Data placed in HTML content between tags (e.g., `<div>USER_INPUT</div>`)
- **HTML_ATTRIBUTE:** Data placed in an HTML attribute (e.g., `<a href="USER_INPUT">`)
- **JAVASCRIPT_STRING:** Data embedded inside a JS string literal (e.g., `var x = 'USER_INPUT';`)
- **URL_PARAM:** Data placed in a URL (e.g., `href="/search?q=USER_INPUT"`)
- **CSS_VALUE:** Data placed in CSS (e.g., `style="color: USER_INPUT"`)

### Step 2: Trace Each Sink Backward
For each sink, trace the data variable backward through the application:

**Early Termination Rule:** When you encounter a sanitizer/encoder, perform two checks:
1. **Context Match:** Is it the correct encoder for this render context? (See Step 5)
2. **Mutation Check:** Any string concatenation or other mutation BETWEEN the sanitizer and the sink?

If correct match AND no intermediate mutations → path is **SAFE**. Stop, document as secure.

**Path Forking:** If a variable can be populated from multiple branches, trace EVERY path independently.

**Track Mutations:** Note all concatenations. A mutation AFTER an encoder invalidates that encoding.

### Step 3: Database Read Checkpoint (Stored XSS)
If backward trace reaches a database read (e.g., `user.find()`, `post.get()`) WITHOUT first hitting
a valid sanitizer, this is a **Stored XSS** vulnerability. Document the specific DB read operation
(file:line, data field accessed). You do not need to trace further back to the write.
A lack of output encoding after a DB read is sufficient to declare the path vulnerable.

### Step 4: Classify the Vulnerability Type
- **Reflected XSS:** Terminates at immediate user input (URL param, form body, header).
- **Stored XSS:** Terminates at a Database Read Checkpoint.
- **DOM-based XSS:** Entire path from source (e.g., `location.hash`, `document.cookie`) to sink
  (e.g., `innerHTML`, `eval`) exists in client-side code only.

### Step 5: Match Encoding to Render Context (Core Rule)
- **HTML_BODY:** Requires HTML Entity Encoding (`<` → `&lt;`, `>` → `&gt;`, etc.)
- **HTML_ATTRIBUTE:** Requires Attribute Encoding (all non-alphanumeric chars HTML-encoded)
- **JAVASCRIPT_STRING:** Requires JavaScript String Escaping (`'` → `\'`, `"` → `\"`, newlines escaped)
- **URL_PARAM:** Requires URL Encoding (`%xx` encoding)
- **CSS_VALUE:** Requires CSS Hex Encoding

**Mismatch:** Path is vulnerable if backward trace completes to a source without matching encoder,
OR if the encoder is correct but applied BEFORE a post-encoding concatenation.

### Step 6: Score Confidence
- **High:** Unambiguous backward trace with clear encoding mismatch, confirmed in code.
- **Medium:** Path plausible but obscured by complex code or minified JavaScript.
- **Low:** Suspicious reflection pattern but backward trace incomplete.
**Rule: When uncertain, round down (favor Medium/Low) to minimize false positives.**

## Advanced Patterns to Check
- **DOM Clobbering:** Can HTML with `id` or `name` attributes overwrite global JS variables?
- **Mutation XSS (mXSS):** Does browser HTML parsing "correct" malformed HTML to create execution?
- **Template Injection:** If server-side templating is used (Jinja2, Handlebars), can template syntax be injected?
- **CSP Bypasses:** What is the CSP? Can it be bypassed via JSONP endpoints, script gadgets in allowed domains, or base-tag injection?

## False Positives to Avoid
- **Self-XSS:** Requires the victim to paste the payload themselves — not a finding.
- **WAF blocking:** Does not mean code is secure; find the underlying flaw.
- **Incorrect encoding as a fix:** HTML-encoding inside a JS string does NOT prevent JS-context XSS.
- **Client-side-only validation:** Not a defense; server-side enforcement is required.

## Browser Verification
Use browser tools to observe live rendering:
- Navigate to endpoints that accept and reflect user input.
- Observe page source and DOM structure for unescaped user-controlled content.
- Check response headers for CSP, X-XSS-Protection, and cookie flags.
- Do NOT inject live XSS payloads — observe structural rendering patterns only.

## Hard Boundaries
- Analysis only — do not steal cookies, session tokens, or exfiltrate data.
- Do not test XSS via stored payloads that would affect other users.
- Stay within authorized scope from the intake report.

## Output Contract

Produce a complete AnalyzerReport with:
- **findings:** Each finding requires: vulnerability type (Reflected/Stored/DOM), complete source-to-sink path,
  render_context label, encoding_observed (file:line), mismatch_reason, evidence (witness payload showing
  render context control), DB read location (for Stored XSS).
- **dominant_patterns:** Named recurring patterns (e.g., "Pattern 1: Missing CSP — inline scripts allowed")
- **strategic_context:** CSP policy analysis, session cookie flags (HttpOnly/Secure), templating engine,

- **secure_vectors:** Confirmed-safe render paths with correct context-appropriate encoding verified
- **warnings:** Minified JS limiting analysis, untraced client-side bundles, CSP bypass assumptions
