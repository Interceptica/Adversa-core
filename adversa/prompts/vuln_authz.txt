## Methodology: Backward Guard Tracing

**Core Principle:** An endpoint is VULNERABLE if a side effect (unauthorized read, mutation, or
privileged action) is reachable without passing through a sufficient authorization guard.
A guard is only valid if it **dominates** all code paths leading to the side effect — guards
placed after a side effect do not count.

### Category 1: Horizontal Authorization Analysis

For each item in Recon Section 8 "Horizontal":

**Trace:** Start at the endpoint. Trace backward through code until you encounter either:
- A **sufficient horizontal guard**, or
- A **side effect** reached without a sufficient guard.

**Definition — Side Effect (horizontal):** Any unauthorized read or mutation of resources belonging
to another user or tenant, including:
- Database read/write of user/org objects belonging to another subject,
- File or config changes affecting another user's data,
- Cross-tenant data access or modification,
- Shared resource leakage between tenants (cache, notifications, logs),
- Metadata exposure across tenant boundaries.

**Sufficient Guard Criteria (horizontal) — ALL must hold:**
- Enforces session authentication,
- Binds the requested resource identifier → current authenticated user/org (ownership check),
- Validates tenant/organization membership in multi-tenant architectures,
- Database queries include proper user/tenant/org scoping (not just the ID from the request),
- Prevents manipulation of user/tenant/org identifiers in request parameters,
- Runs before any side effect,
- Dominates all code paths to the side effect.

**Termination:**
- **GUARDED:** Sufficient guard found before any side effect — document as secure vector.
- **VULNERABLE:** Any side effect reached before a sufficient guard — document as finding.

---

### Category 2: Vertical Authorization Analysis

For each item in Recon Section 8 "Vertical":

**Trace:** Start at the endpoint. Trace backward through code until you encounter either:
- A **sufficient role/capability guard**, or
- A **privileged side effect** reached without one.

**Definition — Side Effect (vertical):** Privileged operations such as:
- User or role management (promote, demote, delete user),
- System configuration changes,
- Database export/import/backup,
- Bulk data access across users,
- Other cross-user or global state changes reserved for privileged roles.

**Sufficient Guard Criteria (vertical) — ALL must hold:**
- Explicit check for the required role or capability (not just authentication),
- Executed before any privileged side effect,
- Dominates all code paths,
- Uses correct guard (not a missing/wrong import, not a no-op stub).

**Termination:**
- **GUARDED:** Sufficient role check dominates sink — document as secure vector.
- **VULNERABLE:** Any privileged side effect occurs before such a guard — document as finding.

---

### Category 3: Context / Workflow Authorization Analysis

For each item in Recon Section 8 "Context":

**Trace:** Walk **forward** through the intended workflow, checking at each step that later actions
validate prior state before allowing the side effect.

**Definition — Side Effect (context):** Workflow-sensitive actions such as:
- Payment capture or financial operations,
- Order/booking confirmation or finalization,
- Account deletion/approval flows,
- Installation/setup completion,
- Status transitions that trigger irreversible actions.

**Sufficient Guard Criteria (context) — ALL must hold:**
- Each workflow step enforces that prior steps completed (via status flags, stage tokens, nonces),
- Guard runs **before** applying the state change,
- Guard cannot be bypassed by skipping directly to a later endpoint.

**Termination:**
- **GUARDED:** All later steps validate prior state before side effects.
- **VULNERABLE:** Any step allows a side effect without confirming prior step completion.

---

### Proof Obligations (apply to all three categories)

- A finding is **GUARDED** if the guard dominates the sink (runs before it on all paths).
- A finding is **VULNERABLE** if a side effect is reachable before a sufficient guard.
- Guards placed **after** the side effect do not count.
- **UI-only checks** (hidden links, disabled buttons, client-side role checks) do NOT count as guards.
- Framework defaults do not provide authorization unless explicitly configured and verified in code.

---

### Confidence Scoring

- **High:** Guard is clearly absent or misplaced in code. Side effect is unambiguous. Path from
  endpoint to side effect is direct with no conditional branches that might add protection.
- **Medium:** Some uncertainty exists — possible upstream controls, conditional logic that might
  add a guard, or the side effect requires specific conditions to trigger.
- **Low:** Vulnerability is plausible but unverified. Multiple assumptions required, unclear code
  paths, or potential alternate controls exist.
**Rule: When uncertain, round down (favor Medium/Low) to minimize false positives.**

## False Positives to Avoid

- **UI-only checks:** Hidden buttons, disabled forms, or client-side role checks do NOT count as
  authorization guards. Server-side enforcement is required.
- **Guards after side effects:** A guard that runs after database writes or state changes does not
  protect those side effects.
- **Confusing auth with authz:** Being authenticated doesn't mean ownership/role checks exist.
  Trace explicitly to find the authz guard.
- **Documentation as proof:** Do not treat policy docs, config comments, or architecture docs as
  evidence of controls. Require code evidence.
- **Business logic confusion:** Intended privilege differences (admins have more access) are not
  flaws — only insecure implementation is.
- **Trusting framework defaults:** Don't assume a framework provides authorization unless explicitly
  configured and verified in the codebase.
- **Missing the side effect:** Focus on where data is actually accessed/modified, not just the
  endpoint entry point. Check indirect access via related objects.
- **Missing tenant validation:** Don't assume tenant isolation exists without explicit checks in
  code. Verify tenant boundaries are enforced in queries.
- **Stopping at insufficient middleware:** Trace all the way to the side effect. Middleware might
  only authenticate, not authorize.
- **Missing state in context flows:** For workflow endpoints, check that EVERY step validates prior
  state, not just the first step.

## Browser Verification

Use browser tools to observe live authorization behavior safely:
- Navigate to protected resource endpoints to confirm auth enforcement.
- Observe HTTP response codes for access control (401 vs 403 vs 200).
- Check if error responses leak object existence (403 = forbidden vs 404 = not found — may reveal
  cross-user resource existence).
- Do NOT attempt to access other users' data with real requests or real sessions.

## Hard Boundaries

- Do not test authorization with real credentials or live sessions.
- Do not enumerate object IDs or perform horizontal sweep attacks.
- Stay within authorized scope from the intake report.
